Froggeric fixed template vs. built-in

Does froggeric's chat_template.jinja (qwen3.6-froggeric-v21.3, XML tool format, as shipped) beat the stock GGUF template for our Qwen3.6 fleet? Two rounds: capability (matched A/B on short-context benches) and the claims themselves (multi-turn KV-cache, token footprint, robustness — tested by rendering, deterministic).

Verdict

Fixes one real thing — at a token cost.

Capability (round 1): no material difference — small-N flips that cancel, built-in already at ceiling. Claims (round 2): it genuinely fixes multi-turn KV-cache destruction, but is ~24% heavier and its other advertised fixes don't reproduce as-shipped.

What this run can’t tell you

  • ceiling

    The stock template already tops out. Toolcalling 10/10, struct-output 100%, instruction-following unchanged on every model — there's almost no headroom for a template to demonstrate a win.

  • scope

    The headline claims live off-frame. KV-cache invalidation from mutated past turns, long-context token waste, and agentic stalls in long loops are multi-turn / long-context behaviors. This suite is short-context and short-turn (by request), so it neither exercises nor refutes them.

  • format

    XML tool calls parsed cleanly. Shipped as Qwen3-Coder XML (<function=…>) rather than the models' native Hermes JSON — llama.cpp still extracted structured tool_calls, so the format swap was not a parser regression at these sizes.

  • power

    Small N, greedy decode. agentic = 5 tasks (one = 20%), reasoning = 12 cases (one = 8.3%). Treat every arrow below as directional, not significant.
